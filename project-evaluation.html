一句话判断

这不是“刚把画布塞进游戏引擎”，而是已经把领域逻辑、显示层、场景、UI 和运行时服务拆开的中后段迁移项目。

1 项目进度

旧版原型仍在根目录保留，新版 Phaser 项目在 phaser/ 中。引擎版能构建，核心循环和多场景结构已落地。

2 改造质量

分层方向正确，GameScene 已经压到约 263 行，说明“大文件拆分”不是停留在文档层。

3 下一阶段风险

大量模块仍通过 scenes/Game.js 门面反向取状态和工具函数，E2E 测试依赖也未写进 package。

当前结构

外层是原始静态原型，phaser/ 是引擎改造版。新版已经采用 Vite + Phaser 4，并注册了 Boot、Menu、Game、Pause、Backpack、Quest、Shop、Forge、Magic 等 Scene。

改造完成度

✓ 已经做得好的地方

从单文件拆成了模块；输入、计时器、显示同步、战斗、AI、任务、经济、背包、魔法等逻辑都有明确归属。

↔ 处在过渡态的地方

registry、事件总线和多 Scene 已接入，但很多调用仍靠共享 state 与 Game.js 门面维系，说明还在迁移桥阶段。

! 不宜继续拖延的地方

如果继续堆功能，state 与 Game.js 门面会重新变成新的中心化负担，后面会越来越难拆。

主要风险

高 测试链不自包含

文档说 E2E 是稳定门，但依赖缺失让 npm test 在干净环境不可用。这会削弱回归保护。

中 分层规则尚未执行到底

架构文档禁止 domain import scenes，但实际仍存在不少反向依赖。当前更像“模块化单体”，还不是干净分层。

中 状态所有权模糊

Phaser registry 已经做了桥，但保存、实体列表、地图和玩家字段仍大多以共享 state 为准。

建议路线

1

先修测试链

把 Playwright 写进 devDependencies，补一个自动启动 Vite 的 E2E runner。目标是 npm test 在干净 clone 后能直接跑。

2

拆掉 Game.js 反向门面

把常量迁到 config，状态从 runtime/state 直接读，服务从 runtime/services 读，UI helper 从 ui 模块读。先从 display 和 ui 开始，最后处理 domain。

3

收口状态所有权

把玩家标量继续迁到 registry，把场景内实体、地图、对象列表迁到 Scene 或专门 world store，保存层只序列化明确快照。

4

再进入玩法深化

在测试链和状态边界稳定后，再扩展剧情、地图、宠物、经济、技能和战斗表现，避免在未收口架构上继续加重量。
